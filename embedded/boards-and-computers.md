# 임베디드에서 자주 쓰는 보드와 컴퓨터

## 1. 한 줄 요약

임베디드 개발에서 쓰는 장치는 크게 마이크로컨트롤러 개발 보드, 싱글 보드 컴퓨터, 산업용 임베디드 PC, SoM, FPGA/SoC 보드로 나눌 수 있으며, 필요한 성능과 실시간성, 전원, 입출력, 운영체제 여부에 따라 선택한다.

## 2. 왜 필요한가?

아두이노와 라즈베리파이는 모두 임베디드 개발에서 자주 보이지만 성격이 다르다. 아두이노는 작은 제어 작업에 맞는 마이크로컨트롤러 보드이고, 라즈베리파이는 리눅스를 실행하는 작은 컴퓨터에 가깝다.

처음에는 둘을 같은 종류로 생각하기 쉽지만, 실제 프로젝트에서는 다음 질문에 따라 선택이 달라진다.

- 운영체제가 필요한가?
- GPIO를 얼마나 정확한 시간에 제어해야 하는가?
- 센서, 모터, 릴레이 같은 주변 회로를 직접 제어해야 하는가?
- 카메라, 네트워크, 파일 시스템, 데이터베이스가 필요한가?
- 배터리로 오래 동작해야 하는가?
- 산업 환경에서 장시간 안정적으로 동작해야 하는가?
- 양산 제품인지, 학습용인지, 프로토타입인지?

## 4. 마이크로컨트롤러 개발 보드

마이크로컨트롤러 개발 보드는 작은 MCU를 쉽게 실험할 수 있도록 만든 보드이다. 일반적으로 리눅스 같은 범용 운영체제를 실행하지 않고, 작성한 펌웨어가 MCU 위에서 직접 동작한다.

### 대표 예

- Arduino Uno, Nano, Mega
- ESP32, ESP8266
- STM32 Nucleo, STM32 Discovery
- Raspberry Pi Pico
- Teensy
- AVR, PIC, MSP430 계열 보드

### 주로 쓰는 곳

- LED, 버튼, 릴레이, 모터 제어
- 온도, 습도, 조도, 거리 센서 읽기
- UART, I2C, SPI 장치 제어
- 간단한 제어기
- 배터리 기반 저전력 장치
- 실시간성이 필요한 간단한 하드웨어 제어

### 장점

- 부팅이 빠르다.
- 전력 소모가 작다.
- GPIO, ADC, PWM, 타이머 제어가 쉽다.
- 하드웨어 동작을 이해하기 좋다.
- 단순한 제어 작업에 적합하다.

### 한계

- 메모리와 저장 공간이 작다.
- 복잡한 UI, 영상 처리, 데이터베이스, 웹 서버에는 불리하다.
- 네트워크나 파일 시스템이 필요하면 구현 부담이 커질 수 있다.
- 운영체제 수준의 기능을 기대하면 안 된다.

## 5. 싱글 보드 컴퓨터, SBC

SBC는 CPU, RAM, 저장장치 인터페이스, 네트워크, USB, HDMI 등을 한 장의 보드에 담은 작은 컴퓨터이다. 보통 리눅스를 실행하며, 임베디드 시스템의 제어기나 게이트웨이로 자주 사용한다.

### 대표 예

- Raspberry Pi
- BeagleBone Black
- Orange Pi
- Banana Pi
- Odroid
- NVIDIA Jetson Nano, Orin Nano
- Rock Pi

### 주로 쓰는 곳

- IoT 게이트웨이
- 카메라, 영상 처리
- 간단한 서버
- 데이터 수집 장치
- 리눅스 기반 장비 제어
- Python, Node.js, C++ 애플리케이션 실행
- AI 추론, 이미지 인식, 로봇 상위 제어

### 장점

- 리눅스와 일반 소프트웨어 생태계를 사용할 수 있다.
- 네트워크, 파일 시스템, USB 장치 사용이 쉽다.
- 고수준 언어와 라이브러리를 쓰기 좋다.
- 카메라, 디스플레이, 웹 서버 같은 기능을 붙이기 쉽다.

### 한계

- 부팅 시간이 필요하다.
- 전원 차단에 취약할 수 있다.
- 정확한 실시간 GPIO 제어에는 부적합할 수 있다.
- 소비 전력이 MCU보다 크다.
- SD 카드 저장장치를 쓰는 경우 파일 시스템 손상을 고려해야 한다.

## 6. 산업용 임베디드 PC

산업용 임베디드 PC는 공장, 설비, 차량, 옥외 환경처럼 장시간 안정성이 필요한 곳에서 쓰는 컴퓨터이다. 일반 PC보다 넓은 온도 범위, 진동, 먼지, 전원 노이즈, 장기 공급 가능성 등을 더 중요하게 본다.

### 대표 형태

- Fanless Embedded PC
- DIN rail PC
- Panel PC
- Rugged PC
- 산업용 게이트웨이

### 주로 쓰는 곳

- 공장 자동화
- 설비 모니터링
- 머신 비전
- HMI
- 데이터 로거
- 산업용 네트워크 게이트웨이

### 장점

- 장시간 동작과 설치 환경을 고려한 설계가 많다.
- 케이스, 전원 입력, 접지, 방열이 제품화되어 있다.
- RS-232, RS-485, CAN, Ethernet 같은 산업용 인터페이스를 제공하는 경우가 많다.
- 유지보수와 현장 설치가 쉽다.

### 한계

- 가격이 높다.
- 크기와 소비 전력이 커질 수 있다.
- 특정 제조사와 모델에 의존하기 쉽다.
- 학습용으로는 과할 수 있다.

## 7. SoM, System on Module

SoM은 CPU, RAM, Flash, PMIC 같은 핵심 부품을 작은 모듈로 만든 것이다. 실제 제품에서는 SoM을 별도의 캐리어 보드에 꽂아 사용한다.

### 대표 예

- Raspberry Pi Compute Module
- NVIDIA Jetson module
- NXP i.MX 계열 SoM
- Toradex Verdin, Colibri
- Variscite, Compulab 계열 모듈

### 주로 쓰는 곳

- 양산 제품의 메인 컴퓨팅 모듈
- 커스텀 I/O가 필요한 리눅스 기반 장비
- 카메라, 디스플레이, 네트워크 장비
- 장기 공급과 인증 부담을 줄이고 싶은 제품

### 장점

- CPU와 메모리 주변의 복잡한 설계를 줄일 수 있다.
- 캐리어 보드만 프로젝트에 맞게 설계하면 된다.
- 양산 제품에 적용하기 좋다.
- 모듈 업체의 BSP, Linux 이미지, 문서를 활용할 수 있다.

### 한계

- 캐리어 보드 설계가 필요하다.
- 모듈 공급 안정성과 가격을 확인해야 한다.
- 업체가 제공하는 BSP와 커널 버전에 의존할 수 있다.
- 단순 학습용으로는 SBC보다 진입 장벽이 높다.

## 8. FPGA / SoC 개발 보드

FPGA 보드는 소프트웨어가 아니라 하드웨어 논리 회로를 구성해서 동작시키는 보드이다. FPGA와 CPU가 함께 들어간 SoC 보드는 고속 신호 처리와 임베디드 리눅스 기능을 함께 사용할 수 있다.

### 대표 예

- Xilinx Zynq 계열 보드
- Intel Cyclone 계열 보드
- Digilent Zybo
- DE10-Nano

### 주로 쓰는 곳

- 고속 병렬 신호 처리
- 영상, 통신, 계측 장비
- 커스텀 디지털 로직
- 정밀한 타이밍 제어
- CPU만으로 처리하기 어려운 하드웨어 가속

### 장점

- 병렬 처리와 정밀한 타이밍 제어에 강하다.
- 원하는 디지털 회로를 직접 구성할 수 있다.
- CPU, MCU보다 낮은 지연 시간으로 동작을 만들 수 있다.

### 한계

- 학습 난이도가 높다.
- Verilog, VHDL, 타이밍 제약, 합성, 배치 배선 개념이 필요하다.
- 단순 제어 작업에는 과한 선택일 수 있다.

## 10. 선택 기준

### 단순 센서와 액추에이터 제어

버튼, LED, 릴레이, 간단한 센서처럼 하드웨어를 직접 제어하는 작업은 MCU 보드가 적합하다.

```text
추천 후보: Arduino, ESP32, STM32, Raspberry Pi Pico
```

### 네트워크와 파일 시스템이 필요한 경우

로그를 파일로 저장하거나 웹 서버, 데이터베이스, 카메라, USB 장치를 사용해야 한다면 SBC가 편하다.

```text
추천 후보: Raspberry Pi, BeagleBone, Odroid, Orange Pi
```

### 영상 처리나 AI가 필요한 경우

카메라 영상 처리, 객체 인식, 딥러닝 추론이 필요하면 GPU나 NPU가 있는 보드를 검토한다.

```text
추천 후보: NVIDIA Jetson 계열, AI 가속기가 있는 SBC
```

### 산업 현장에 설치해야 하는 경우

24V 전원, 노이즈, 접지, 장시간 동작, 케이스, 냉각, 통신 포트가 중요하면 산업용 PC나 산업용 게이트웨이를 검토한다.

```text
추천 후보: Fanless PC, DIN rail PC, 산업용 게이트웨이
```

### 양산 제품을 만들 경우

프로토타입은 SBC로 시작하더라도, 제품화 단계에서는 SoM이나 직접 설계한 MCU 보드가 더 적합할 수 있다.

```text
추천 후보: SoM + Carrier board, 커스텀 MCU 보드
```

### 정밀한 타이밍이나 고속 병렬 처리가 필요한 경우

CPU나 MCU가 순차적으로 처리하기 어려운 고속 신호 처리에는 FPGA를 검토한다.

```text
추천 후보: FPGA 보드, FPGA + CPU SoC 보드
```

## 11. 자주 하는 오해

- 아두이노와 라즈베리파이는 같은 종류가 아니다. 아두이노는 보통 MCU 개발 보드이고, 라즈베리파이는 보통 리눅스 SBC이다.
- MCU가 SBC보다 항상 하위 선택지는 아니다. 실시간 제어, 저전력, 단순 제어에서는 MCU가 더 적합하다.
- 라즈베리파이 GPIO로 모든 하드웨어를 직접 구동하면 안 된다. 전압, 전류, 보호 회로, 레벨 변환을 확인해야 한다.
- 리눅스가 올라가는 보드는 전원 차단과 파일 시스템 손상 문제를 고려해야 한다.
- 개발 보드에서 동작했다고 바로 양산 가능한 것은 아니다. 전원, 보호, EMI, 온도, 커넥터, 장기 공급을 다시 검토해야 한다.
- 보드의 GPIO 전압이 3.3V인지 5V인지 반드시 확인해야 한다.
- 모터, 릴레이, 솔레노이드 같은 부하는 보드 핀에 직접 연결하지 않는다.

## 12. 실무 체크리스트

- [ ] 운영체제가 필요한 작업인가?
- [ ] 정확한 실시간 제어가 필요한가?
- [ ] GPIO 전압 레벨이 주변 회로와 맞는가?
- [ ] 보드 핀의 최대 전류를 넘지 않는가?
- [ ] ADC, PWM, UART, I2C, SPI, CAN 등 필요한 주변장치가 충분한가?
- [ ] 전원 전압, 최대 소비 전류, 순간 피크 전류를 계산했는가?
- [ ] 전원 차단 시 데이터 손상 가능성을 검토했는가?
- [ ] 발열과 케이스 내부 온도를 고려했는가?
- [ ] 장기 공급 가능성과 대체 부품을 확인했는가?
- [ ] 개발용 보드와 실제 제품용 보드를 구분해서 판단했는가?

## 13. 기억할 문장

아두이노, 라즈베리파이, 산업용 PC는 모두 임베디드 개발에서 쓰일 수 있지만 같은 문제를 푸는 도구는 아니다. 작은 하드웨어를 정확하게 제어해야 하면 MCU 보드가 유리하고, 리눅스 기능과 네트워크, 파일 시스템, 카메라가 필요하면 SBC가 유리하다. 현장 설치와 장기 안정성이 중요하면 산업용 PC나 SoM 기반 설계를 검토해야 한다.
